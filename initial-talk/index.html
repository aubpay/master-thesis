Automatic Universal In-Browser Payments

An alternative model to web advertising

Daan Middendorp ‐ November 6th, 2019

Problem & Statement


Unpaid online content:
  • Targeted advertising
  • Subscriptions
  • Microtransactions

  • →  Privacy invasive
  • →  Vendor lock-in
  • →  Too much effort / vendor lock-in

Targeted advertising

Micropayments / subscriptions

Related work

Academic:
  • Privad
  • Brave browser

The brave browser provides an example of a system where it is possible to donate to publishers. However, this system is still suboptimal, because you need to install a different browser, which requires a lot of effort. Furthermore, their business model is questionable, some publishers are complaining that it is really hard to receive your money and their are taking away ad revenue by blocking ads by default. Or (in some cases) replacing the ads of a publisher with their own.

Desired solution
  1. User requests content
  2. Handshake is performed
  3. Is there money in the wallet?
  4. If yes: pay, show ads otherwise
It is important to understand that this system works as an alternative to advertisment. However, it is up to the user. He can decide to pay or not to pay. The consequence will be that ads are shown.

Possible strategies

  1. Amount x per hour
  2. Amount y per view with maximum amount per publisher/hour
  3. Amount z per publisher per day
The problem with this approach is that we need a fair way to spend the money to the different publishers. In order to do this, we came up with diferent strategies.

Requirements

  1. as-easy-to-use as ads
  2. HTTP and JS-based
  3. No vendor lock-in
  4. Not privacy invasive
  5. Open source / transparent
With the new possiblities that came with the invention of blockchain technology, it becomes possible to develop a system with these requirements.

Challenges

  • How to request a payment?
  • How to keep it decentral?
  • Where to store the wallet?
  • How to keep it private and transparent?
  • ...

Evaluation Proof-of-Concept

  • Automated payments
  • No configuration
  • No single authority
  • Tamper proof

Conclusion

  • Automatic Universal In-Browser Payments
  • An alternative model to web advertising
  • Based on crypto
  • No single authority
